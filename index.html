一路以来的成长

这条路不是一开始就规划好的。很多能力都是在一次次发版、回归、工单和环境问题里慢慢长出来的：先学会把需求和用例写清楚，再理解云平台的真实风险，再把重复验证交给脚本。到 Q1，我开始更明确地把测试往前推，在需求评审阶段拆风险、补边界，让问题尽量在编码和提测前暴露。

Stage 01

先把测试基本功做扎实

从 PRD、Wiki、截图和工单中拆验收点，写用例、记录缺陷、复现问题、跟进回归，逐步形成“过程要留痕、边界要清、结论要可追溯”的习惯。

沉淀 6000+ 用例/检查项，覆盖 400+ 文档/表格节点。
Stage 02

进入云平台真实复杂度

在 Sealos 公有云和私有化交付中，测试对象从单点功能扩展到 DevBox、数据库、应用商店、Admin、计费、GPU、网络和存储等跨模块链路。

2025 Q3/Q4 记录并跟进 900+ 缺陷。
Stage 03

把重复验证变成工具能力

开始用 Playwright、TypeScript、Shell 和 kubectl 做真实 UI 回归、底层状态检查、资源清理和报告输出，让回归不只依赖人工记忆。

部分回归从约 4h/轮降至 2.4h/轮。
Stage 04

开始把测试往前推

在需求评审阶段提前介入，围绕订阅制、私有化、GPU 监控等任务拆场景和边界。AI 和 Skill 更多用来辅助需求建模、风险分析和测试工件生成，而不是替代判断。

Q1 回归耗时降低约 40%，并提前覆盖多个边缘场景。

现在的一个高点

Q1 对我来说是一个明显的节点。以前更多是在提测后找问题，现在开始把测试放到需求和设计阶段：提前澄清模糊点，拆组合边界，推动用例评审，把线上问题反哺到检查清单和自动化回归里。这也是我理解的测试左移。

  • 在需求阶段介入，提前识别订阅制、私有化、镜像同步等场景的组合边界。
  • 用例和回归流程更规范，部分回归从约 4h/轮降到 2.4h/轮。
  • 完成 17 项测试任务，覆盖私有化发版、订阅制、多卡训练、GPU 监控等高风险任务。
  • 继续补质量门禁、自动化稳定性、云原生排障和跨团队质量推进经验。

FROM UI SYMPTOM TO ROOT CAUSE

UI 回归结合 Kubernetes 状态检查

云平台测试让我真正理解，测试不能只停在页面现象。很多问题表现在 UI 上，根因却在 Pod、镜像、网络、存储、Runtime 或环境差异里。我开始把页面路径、接口返回、资源状态和日志放在一起判断。

变化从“描述页面异常”走向“定位链路和环境问题”。
行动检查 Running/Ready、kubectl exec、SSH runtime、公网入口、端口连通性和资源清理。
价值把测试结论从“页面正常”推进到“链路和资源状态可用”。

FROM PROMPT TO LEFT SHIFT

Test Skill 与测试左移探索

我尝试 AI 测试时，很快发现“让 AI 直接生成用例”并不可靠。真正有价值的是把测试思考拆成流程：先做需求澄清和风险判断，再产出测试/诊断工件，条件满足后再进入执行或诊断结论输出。

变化从“用 AI 生成内容”走向“用 AI 辅助需求建模和测试左移”。
行动识别任务类型、选择 truth model、做需求澄清和 blocker 判断，再产出测试/诊断工件。
价值把测试设计往需求阶段前移，让关键 case 更早进入验收和回归。
